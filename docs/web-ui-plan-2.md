# Web UI 고도화 계획 (2차) — 보안, 읽기 히스토리, 발견

> [web-ui-plan.md](web-ui-plan.md)(M1–M4, 완료)의 후속. 실행 보드는
> [web-ui-board.md](web-ui-board.md)의 M5–M8.
>
> 관통하는 주제: **위키가 먼저 말을 걸게 한다**(philosophy.md 원칙 6의 웹 실현).
> M1–M4가 "위키를 읽고 쓰는 곳"을 만들었다면, 2차는 위키가 사용자에게
> 안 읽은 것·잊은 것·연결 안 된 것·비어 있는 것을 능동적으로 보여주는 단계다.

## 1. 문제 정의

1. **serve가 모든 인터페이스에 바인딩된다.** 인증이 없으므로 실수로 공용망에
   노출되면 위키 전체(편집 포함)가 열린다. 기본값이 안전해야 한다.
2. **위키는 내가 뭘 읽었는지 모른다.** "새발견"(안 읽은 페이지 추천)을 하려면
   열람 이력이 필요한데, 지금은 신호가 없다. 스쳐 지나간 방문을 "읽음"으로
   오인하면 추천이 망가지므로 명시적 표시와 신중한 자동 감지가 함께 필요하다.
3. **링크는 명시적 교차링크뿐이다.** 작성 시점에 안 만든 연결은 영영 보이지
   않는다. 임베딩이 이미 있으므로 페이지마다 "느슨한 제안 링크"를 보여줄 수 있다.
4. **재발견 루프가 웹에 없다.** resurface/bridge는 에이전트 경유(Telegram)로만
   동작한다. 매일 여는 화면(웹 UI)이 그 루프의 1차 접점이 되는 게 자연스럽다.

## 2. 설계 결정

### D1. 기본 바인딩은 localhost, 공개 바인딩은 인증 필수 (fail-safe)

- `--addr` 기본값을 `:8737` → `localhost:8737`로 변경.
- loopback 밖 인터페이스로 바인딩하려는데 인증이 설정되어 있지 않으면
  **기동을 거부**하고 설정 절차를 안내한다. "공개인데 무인증" 상태는
  만들 수 없게 한다.

### D2. 부트스트랩 인증 — 터미널 1회용 설정 코드

"최초 로그인 때 id/pw를 정하게" 하는 방식의 알려진 함정: 공개 바인딩 직후
크리덴셜이 정해지기 전 창에 **공격자가 먼저 계정을 등록**할 수 있다(선점 레이스).
이를 막는 부트스트랩:

1. 인증이 필요한데 계정이 없으면, serve가 **1회용 설정 코드를 터미널에만 출력**한다.
2. `/setup` 페이지에서 그 코드 + 원하는 id/pw를 입력해야 계정이 만들어진다.
   (터미널을 볼 수 있는 사람 = 운영자만 등록 가능)
3. 이후에는 `/login`으로 id/pw 로그인.

- 저장: `~/.config/canopy/webauth.json`에 **bcrypt 해시**(x/crypto는 이미 의존성).
  위키 저장소 밖이다 — 비밀이 git으로 새지 않고, 위키를 공유해도 계정은 기기별.
- 세션: 서버 메모리의 랜덤 토큰 + `HttpOnly; SameSite=Lax` 쿠키(30일).
  재시작하면 재로그인 — 단일 사용자 도구에서 수용 가능한 단순함.
- 로그인 실패에 지수 지연(브루트포스 완화). 편집 등 POST는 Origin 검사 추가.
- TLS는 구현하지 않는다 — tailscale·Cloudflare Tunnel·리버스 프록시에 위임하고
  문서로 안내한다. (계정 재설정 = webauth.json 삭제 후 재부트스트랩)
- localhost 바인딩일 때는 지금처럼 무인증 — 로컬 사용 경험을 해치지 않는다.

### D3. "읽음"은 명시가 1급, 자동 감지는 보조

- **명시적**: 페이지에 `✓ 읽음` 버튼(+ 단축키 `r`). 한 번 누르면 끝 — 가장 싼 동작.
- **자동 감지**: "어쩌다 들른" 방문을 배제하는 보수적 임계 —
  탭이 보이는 상태의 체류 시간이 본문 길이에 비례한 기준(예: 최소 30초)을 넘고
  **스크롤이 본문 70%를 지나면** 읽음 후보로 기록. `visibilitychange`로 백그라운드
  탭 시간은 제외. 자동 읽음은 `auto`로 구분 표시되고 클릭 한 번으로 취소 가능.
- 저장: `<wiki>/_meta/webui/reads.json` — resurface state와 같은 원칙(재구축 불가
  상태는 위키에 커밋되어 기기 간 동기화). 포맷은 문서화해서 CLI·에이전트도 읽는다:
  `{version, reads: {slug: {first, last, count, source: explicit|auto}}}`
- 페이지 이름 변경(mv)은 slug 기반이므로 이력이 따라가지 못한다 — mv 시 reads 키도
  이관하도록 CLI mv에 훅을 추가한다(같은 writeops 계열 처리).

### D4. 새발견 = "안 읽은 페이지"의 랭킹 추천

unread 페이지를 세 신호로 랭킹한다 (모두 기존 데이터로 계산 가능):

| 신호 | 의미 | 소스 |
|---|---|---|
| 신규성 | 최근 생성됐는데 아직 안 봄 (에이전트가 만든 페이지를 놓침) | created |
| 허브성 | 백링크 많은 중요 페이지인데 안 봄 | scan.Backlinks |
| 관심 인접 | **최근 읽은 페이지들과 시맨틱 유사**한 미독 페이지 | PageVectors × reads |

- 홈에 "새발견" 섹션(3–5개) + 전체 목록 `/special/discover`(facet 필터 가능).
- resurface(한때 알던 것의 재발견)와 상보적 — discovery는 "처음 만나는 것".
  reads 데이터는 이후 resurface의 진짜 열람 신호로도 쓸 수 있다(아래 M8+).

### D5. 느슨한 제안 링크 — 페이지 단위 bridge

- 페이지 하단에 "제안 링크" 섹션: 이 페이지와 시맨틱 유사도가 높지만
  **아직 링크(양방향 모두)가 없는** 페이지 top-N(유사도 표기).
  `store.PageVectors` 코사인 + 태그 겹침 부스트(`rankRelated`와 같은 랭킹 사상),
  임계값은 bridge의 `--min-sim` 기본(0.7)을 따른다.
- 명시적 링크(본문 위키링크·백링크)와 시각적으로 구분한다(점선 테두리 등) —
  "위키가 확언하는 연결"과 "위키가 추측하는 연결"을 섞지 않는다.
- 제안을 본문 링크로 승격하는 것(원클릭 연결)은 M8+ 후보로 미룬다 —
  링크는 콘텐츠 판단이므로 자동 삽입은 신중해야 한다(philosophy 원칙).

### D6. 위키가 먼저 말을 거는 홈

- 홈에 **"오늘의 재발견" 카드 1개**: `resurface.PickPages`(auto 전략)의 pick을
  발췌와 함께 보여주고 👍/👎/😴(스누즈) 버튼을 단다. 버튼은 CLI
  `resurface feedback`과 **같은 state 파일**에 기록 — 웹과 에이전트(Telegram)가
  같은 쿨다운·피드백을 공유하므로 중복 노출이 없다.
- **검색 갭 로그**: 검색 결과가 빈약했던(0건 또는 저점수) 질의를
  `_meta/webui/search-gaps.jsonl`에 기록하고 `/special/gaps`에서 보여준다 —
  "위키가 답하지 못한 질문"은 곧 페이지 생성 후보다(P2 query-filing 루프의
  수요 측 신호). 에이전트가 이 파일을 읽어 페이지 초안을 제안할 수 있다.

## 3. 마일스톤

### M5 — 보안: localhost 기본 + 설정 코드 부트스트랩 인증

기본 바인딩 변경, webauth 저장소, /setup·/login·/logout, 세션 미들웨어,
공개 바인딩 가드, 로그인 지연, POST Origin 검사, troubleshooting 문서.
**Exit**: 공개 바인딩은 인증 설정 없이 기동 불가. 설정 코드 없이 계정 등록 불가.
localhost는 기존처럼 무인증. 인증 후 전 기능(편집 포함) 동작.

### M6 — 읽기 히스토리와 새발견

reads.json 저장소, `✓ 읽음` 버튼+단축키, 체류·스크롤 자동 감지(취소 가능),
mv 시 이력 이관, 홈 새발견 섹션, `/special/discover`, 읽기 진행률 표시.
**Exit**: 명시/자동 읽음이 구분 기록되고 취소 가능. 스쳐간 방문(짧은 체류)은
읽음이 되지 않음을 실측 확인. 새발견 추천에서 읽은 페이지가 즉시 빠진다.
reads.json이 sync로 커밋된다.

### M7 — 느슨한 제안 링크

페이지 하단 제안 링크 섹션(미연결 유사 페이지, 유사도·태그 부스트 랭킹,
명시 링크와 시각 구분), 성능 확인(요청당 벡터 로드 비용).
**Exit**: 이미 링크된 페이지는 제안에 나오지 않는다. 제안 클릭으로 이동 가능.
250페이지 규모에서 페이지 로드 지연 무감지(<50ms 추가).

### M8 — 위키가 먼저 말을 거는 홈

오늘의 재발견 카드(피드백 버튼 → resurface state 공유), 검색 갭 로그와
`/special/gaps`, bridge 제안 카드(보기만, 연결은 CLI/에이전트 안내).
**Exit**: 웹 👍/👎/😴가 CLI `resurface feedback`과 같은 파일에 기록되고
쿨다운이 공유된다. 빈약한 검색이 갭 로그에 쌓이고 화면에서 확인된다.

### M8+ (이번 범위 밖, Icebox)

- reads를 resurface 후보 선정의 진짜 열람 신호로 사용 (지금은 updated 날짜 근사)
- 제안 링크 원클릭 승격 (writeops 경유 본문 링크 삽입)
- digest에 읽기 통계 포함 ("이번 달 몇 페이지를 읽었나")
- TLS 직접 종단 (요구 생기면; 그 전엔 tailscale/프록시 문서로)

## 4. 비범위 / 리스크

- 다중 사용자·권한 분리: 비범위. 계정은 1개(단일 사용자 도구).
- 자동 읽음 감지의 오탐/미탐: 보수적 임계로 시작하고, 명시 버튼이 항상 우선.
  임계값은 상수로 두고 튜닝 여지만 남긴다(설정 항목 추가는 미룸).
- `_meta/webui/` 쓰기는 위키 저장소를 더럽히는 게 아니라 resurface state와 같은
  "동기화가 필요한 상태" 범주다 — invariants.md에 점검 항목을 추가한다.
- 브라우저 이벤트 기반 감지는 JS 없는 환경에서 동작하지 않는다 — 명시 버튼은
  form POST라 JS 없이도 동작하게 만든다.
