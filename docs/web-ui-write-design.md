# Web UI 쓰기(M4) 설계 — 동시성과 충돌 처리

> [web-ui-board.md](web-ui-board.md) M4의 선행 설계 문서.
> 원칙: **우회 쓰기 경로를 만들지 않는다.** 웹 편집은 CLI `update`와
> 동일한 검증·동일한 파이프라인(`writeops.Run`)을 지나야 한다.

## 1. 쓰기 경로 통일

CLI `afterWrite`의 불변 파이프라인(스캔 → genindex → logops → reindex)을
`internal/writeops.Run`으로 추출했다. CLI 명령과 웹 편집 핸들러가 **같은
함수를 호출**하므로 부수효과가 정의상 동일하다 — "웹 편집이 CLI와 같은
효과를 남기는지"는 코드 경로가 하나라는 사실로 보장된다.

웹 편집의 범위는 CLI `update --body-file`과 정확히 같다:
**본문 교체 + `updated` 날짜 갱신.** frontmatter(type/tags/제목)는 웹에서
편집할 수 없다 — 스키마 필드는 검증 규칙이 얽혀 있어(CLI가 이미 전담)
웹 폼에 노출하면 우회 경로가 된다. 페이지 생성/이동/삭제도 CLI 전용 유지.

## 2. 동시성 모델

로컬 단일 사용자 전제(인증 없음, 사설망)에서 동시 쓰기는 두 종류다:

| 시나리오 | 처리 |
|---|---|
| 웹 편집 ↔ 웹 편집 (탭 2개) | 낙관적 잠금(아래) |
| 웹 편집 ↔ CLI/에이전트 쓰기 | 낙관적 잠금이 웹 쪽을 보호. CLI끼리의 경쟁은 지금과 동일(파일시스템 최종 승자 + git 안전망) |

서버 내부에서는 기존 `Server.mu`가 쓰기 핸들러를 직렬화한다
(검색·엔진과 같은 뮤텍스 — 쓰기 빈도가 낮아 경합 무의미).

## 3. 낙관적 잠금 (MediaWiki 편집 충돌 패턴)

1. `GET /edit/{slug}` 폼에 **로드 시점 파일 내용의 SHA-256**을 hidden으로 담는다.
2. `POST /edit/{slug}`에서 현재 파일 해시와 비교.
3. 불일치 → 저장하지 않고 **409 충돌 화면**: 사용자의 편집본(textarea 유지)과
   "다른 곳에서 수정됨" 안내를 보여주고, 새 해시로 다시 제출하게 한다.
   자동 병합은 하지 않는다(단일 사용자에서 충돌은 드물고, 잘못된 자동 병합이
   조용한 데이터 손실보다 위험).

## 4. 저장 전 검증과 lint

- frontmatter가 파싱 가능한 상태로 유지되는지 확인(본문만 교체하므로
  구조적으로 깨질 수 없지만, 방어적으로 `wiki.Parse` 후 `FMErr` 검사).
- 저장 후 `lint.Run`을 돌려 **이 페이지의 finding만** 편집 결과 화면에
  표시한다(끊긴 링크 등). CLI와 동일하게 lint는 보고만 하고 저장을
  막지 않는다 — 막으면 CLI보다 엄격한 별도 규칙이 생기는 것.

## 5. sync는 웹에서 하지 않는다

커밋/push는 의도적 행위로 남긴다(philosophy.md). 편집 후 화면에
`canopy sync` 안내만 표시. 나중에 필요해지면 별도 결정으로.
