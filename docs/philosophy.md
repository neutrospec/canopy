# Canopy 철학

> 이 문서의 규칙: **모든 원칙은 "누가 강제하는가"와 "어떻게 점검하는가"를 함께 적는다.**
> 점검 명령이 없는 주장은 원칙이 아니라 희망사항이므로, 이 문서가 아니라
> [second-brain.md](second-brain.md)의 로드맵에 적는다.
>
> 강제 수준 표기 — **[코드]**: 위반이 불가능하거나 명령이 거부함.
> **[검출]**: 위반은 가능하지만 `canopy lint`가 잡아냄. **[협약]**: 코드가 강제하지
> 않는 약속. 협약은 최소화하고, 협약임을 숨기지 않는다.

## 배경: 왜 이 프로젝트인가

이전 시스템(2026-04~06)은 규칙을 900줄짜리 스킬 문서의 산문 체크리스트로 유지했다.
그 문서에는 실패 사례가 계속 append됐다: index 수치 드리프트, git push 누락,
root 파일 생성, 로그 형식 혼동. **문서가 위반 기록으로 비대해지는 것 자체가
"프롬프트로 규칙을 강제할 수 없다"는 증거였다.**

Karpathy의 LLM wiki 철학에서 Memex가 못 풀었던 문제는 "누가 유지보수를 하는가"였고,
그의 답은 LLM이었다. canopy의 답은 한 단계 더 구체적이다:
**판단은 LLM이, 불변식은 코드가.** LLM에게 체크리스트를 외우게 하지 말고,
체크리스트가 필요 없는 도구를 쥐여준다.

## 원칙

### 1. 규칙은 프롬프트가 아니라 코드가 강제한다 **[코드]**

스키마 위반(잘못된 type, taxonomy 밖 tag, 한글/대문자 파일명, root 생성, 존재하지
않는 페이지로의 링크)은 에이전트가 "안 하기로 약속"하는 게 아니라 `canopy new`가
**거부**한다.

- 점검: `canopy new x --type guide` → 에러. `canopy new x --type concept --tags 없는태그` → 에러.
  `canopy new "제목" --type concept --links no-such-page` → 에러.

### 2. 파생물은 손으로 만지지 않는다 **[코드]**

`index.md`, `index/*.md`, `logs/*.jsonl`, 검색 인덱스 DB(`$XDG_CACHE_HOME/canopy/`)는 canopy가 생성한다.
수치는 항상 파일시스템 실측이므로 "+1 드리프트" 클래스의 버그가 존재할 수 없다.
생성 파일에는 `<!-- generated by canopy -->` 마커가 있다.

- 점검: `canopy status --json`의 pages 수 == `index.md`의 Total pages ==
  `ls entities concepts comparisons | grep -c '\.md$'` 실측값.
- 손으로 고쳐도 다음 쓰기 명령이 전체를 재생성하므로 드리프트가 자가 치유된다.

### 3. 원본과 상태와 캐시를 구분한다 **[코드]**

| 데이터 | 위치 | 성격 |
|---|---|---|
| 지식 (페이지, raw/) | 위키 repo, 커밋 | 원본 — 사람과 LLM의 산출물 |
| 스키마 (canopy.toml) | 위키 repo, 커밋 | 원본 — 데이터의 규칙은 데이터와 함께 여행 |
| resurface 상태 (`_meta/resurface/`) | 위키 repo, 커밋 | **재현 불가** 상태 — 기기 간 동기화 필요 |
| 읽기 이력·검색 갭 (`_meta/webui/`) | 위키 repo, 커밋 | **재현 불가** 상태 — 주의(attention)의 기록도 동기화 필요 (원칙 12) |
| 검색 인덱스 | `$XDG_CACHE_HOME/canopy/index/<해시>.db` | 파생 캐시 — 위키 밖, 언제든 재구축 |
| 웹 인증 (`webauth.json`) | `$XDG_CONFIG_HOME/canopy` | 머신 로컬 **비밀** — git으로 여행하면 안 되는 것 |
| 전역 설정 / 모델 | `$XDG_CONFIG_HOME/canopy` / `$XDG_DATA_HOME/canopy` | 머신 로컬 (XDG Base Directory 준수) |

위키 안에 canopy가 두는 것은 스키마와 두 상태 디렉토리뿐이고, 전부 "위키의
일부여야 하는 이유"(규칙의 동행, 상태의 동기화)가 명확한 것만 남긴 결과다.
반대로 비밀(인증)은 어떤 경우에도 위키에 넣지 않는다 — 위키는 공유·복제되는
물건이기 때문이다. 캐시가 위키 밖이므로 gitignore 항목조차 필요 없다.

- 점검: `rm -rf ~/.cache/canopy && canopy reindex && canopy search "아무거나"` — 손실 없이 복구되면 캐시가 맞다.
- 점검: `test ! -e <wiki>/.canopy` (위키 안에 캐시 없음). `git -C <wiki> ls-files _meta/` → resurface·webui 추적됨.
- 점검: `git -C <wiki> ls-files | grep -c webauth` == 0 (비밀은 위키 밖).

### 4. 검색이 탐색을 대체한다 (The Wall 해법) **[코드]**

Karpathy가 지적한 "index.md가 수백 페이지를 넘으면 탐색 불가" 문제를 index 계층
확장이 아니라 검색(FTS5 BM25 + bge-m3 벡터, RRF 융합)으로 푼다. index.md는
사람용 대시보드일 뿐, 에이전트의 진입점은 `canopy search`이고 사람의 진입점은
웹 UI의 검색박스다(검색-우선 홈, [web-ui-plan.md](web-ui-plan.md) 참조).
분류가 필요한 곳에서도 트리를 키우지 않고 facet(dir×type×tag 교차 필터)을 쓴다.

- 점검: `canopy search "산업용 프로토콜" --json` — 한국어 시맨틱 매칭 확인.
- 임베딩은 프로세스 내장(ONNX Runtime)이다. 외부 서버가 없으므로 "서버 다운 →
  검색 마비" 실패 모드가 없다. 점검: 네트워크 끊고 `canopy search --mode semantic`.

### 5. 커밋은 의도적 행위, 잊어버림은 시스템이 막는다 **[코드]**

여러 쓰기를 하나의 커밋으로 묶는 판단은 사용자/에이전트의 것이므로 자동 커밋하지
않는다. 대신 잊어버림이 침묵하지 못하게 한다: 모든 명령이 시작 시 미동기 배너(⚠)를
띄우고, 쓰기 명령은 `NEXT: canopy sync`로 끝난다. `canopy sync`는 pull을 먼저
수행하므로 "pull 없이 push 충돌" 모드도 없다.

- 점검: 위키 파일 하나 touch → 아무 canopy 명령이나 실행 → stderr에 ⚠ 배너.
- 점검: `canopy sync` 출력에 pull 단계가 항상 선행.

### 6. 코드가 후보를 만들고, LLM이 판단한다 **[코드+협약]**

의미론적 작업(모순 탐지, 연결 제안, 재발견 메시지 작성)은 코드만으로 못 하고,
LLM만으로 하면 재현성이 없다. 분담: canopy가 결정론적 후보를 생성하고
(`canopy bridge`, `canopy resurface` — 유사도·백링크·날짜 기반), 에이전트가
후보를 판단·문장화·전달한다. 에이전트가 후보 없이 임의로 고르는 것은 협약 위반.

- 점검: `canopy bridge --peek --json` — 임계값 이상·미연결 페어만 나오는지.
- 점검: 같은 state에서 `canopy resurface --strategy hub --peek` 반복 실행 → 동일 결과(결정론).

### 7. raw/ 는 불변이다 **[협약]**

canopy는 raw/를 절대 쓰지 않지만(코드에 raw/ 쓰기 경로가 없음), 에이전트나 사람이
고치는 것까지 막지는 못한다. 정정은 위키 페이지에서 하고 raw는 그대로 둔다.

- 점검(부분): `git -C <wiki> log --oneline -- raw/ | head` — raw 수정 커밋이 있는지 감사.

### 8. 문서의 모든 주장은 점검 가능해야 한다 **[협약 — 이 문서 자신에 적용]**

이 문서와 [invariants.md](invariants.md)에 점검 명령 없는 주장을 추가하지 않는다.
새 기능은 (1) 불변식을 정하고 (2) 점검 방법을 invariants.md에 먼저 적은 뒤 구현한다.
전체 감사는 invariants.md의 절차로 언제든 수행할 수 있다.

### 9. 문은 여러 개여도 복도는 하나다 **[코드+협약]**

쓰기 인터페이스는 늘어날 수 있다 — CLI, 웹 편집기, 앞으로 무엇이든. 원칙은
"CLI만 쓴다"가 아니라 **모든 변경이 같은 파이프라인(`writeops.Run`: 검증 →
쓰기 → index 재생성 → 로그 → 재인덱싱)을 지난다**는 것이다. 기준은 프로그램
소속이 아니라 파이프라인 동일성이다: canopy 안에 있어도 이 복도를 우회하는
쓰기 코드는 위반이고, 새 인터페이스가 복도를 지나면 원칙은 지켜진 것이다.
웹 편집기는 이 이유로 CLI `update`와 문자 그대로 같은 함수를 호출하며, 노출
범위를 본문 교체로 좁힌 것도 스키마 검증을 우회할 폼을 만들지 않기 위해서다
([web-ui-write-design.md](web-ui-write-design.md)).

경계의 정직한 표기: 이 원칙이 [코드]인 것은 **canopy가 제공하는 문**에
한해서다. Obsidian이나 에이전트가 파일을 직접 고치는 옆문은 코드로 막을 수
없다 — 그쪽은 [협약](canopy 명령을 거치게 하라) + 자가치유(파생물은 다음 쓰기
때 전체 재생성) + lint 검출로 대응한다.

- 점검: 스크래치 위키에서 웹 편집과 `canopy update`를 나란히 실행 → 파일
  (updated 갱신)·logs 엔트리·index/·FTS 결과가 동일 (invariants I2).
- 점검(코드 감사): `grep -rn "os.WriteFile" internal/webui/ | grep -v _test`
  → 정확히 두 곳: 편집 저장(edit.go — 직후 `writeops.Run` 호출)과 인증
  파일(auth.go — 위키 밖 `webauth.json`). 그 외에 위키 페이지를 쓰는 곳이
  생기면 이 원칙의 위반이다.

### 10. 확언과 추측을 구분한다 **[코드]**

위키가 아는 것에는 두 등급이 있다. **확언**: 사람/에이전트가 판단해 심은 명시적
위키링크. **추측**: 임베딩 유사도가 발견한 후보(페이지 하단 제안 링크, bridge
페어, 섬의 다리 제안). canopy는 추측을 확언으로 자동 승격하지 않고 — 링크
삽입은 콘텐츠 판단이므로 원칙 6의 분담을 따른다 — UI에서도 둘을 시각적으로
구분한다(제안은 점선). 추측이 본문 링크와 겹치면 후보에서 제외되므로 같은
연결이 두 등급에 동시에 존재하지 않는다.

- 점검: `canopy bridge --peek --json` → 전부 `linked == false`.
- 점검: 웹 페이지 하단 제안 링크에 그 페이지의 아웃바운드/백링크가 나오지 않는다.

### 11. 공개는 곧 인증이다 **[코드]**

serve의 기본 바인딩은 localhost다. loopback 밖으로 여는 순간 인증 벽이 자동
활성되며, 계정이 없으면 터미널에만 출력되는 1회용 설정 코드 없이는 계정을
만들 수 없다(선점 레이스 차단). "공개인데 무보호"인 상태는 설정 실수로도
만들 수 없어야 한다 — 안전은 기본값의 성질이지 사용자의 주의력이 아니다.

- 점검: `canopy serve --addr :8737` 기동 후 무인증 `curl -s -o /dev/null -w '%{http_code}' http://<ip>:8737/` → 302 (내용 없음).
- 점검: `curl localhost:8737` (localhost 바인딩) → 200, 인증 없음 그대로.

### 12. 주의(attention)도 상태다 **[코드]**

무엇을 읽었는가는 재현 불가능한 상태이므로 위키에 커밋되어 동기화된다
(`_meta/webui/reads.json`). 기록의 등급: **명시(✓ 읽음 버튼)가 1급**이고 자동
감지(체류+스크롤)는 보조이며, 자동은 명시를 절대 강등하지 못하고 언제든 취소
가능하다. 스쳐간 방문을 읽음으로 오인하지 않는 보수성이 추천(새발견)의 품질을
지킨다. 같은 논리로 검색 갭(위키가 답 못한 질의)도 상태로 축적된다.

- 점검: invariants H1–H3.

### 13. 연결성은 도달 가능성까지다 **[검출]**

고아 검사(백링크 0)는 필요조건일 뿐이다. 서로는 링크됐지만 본토(최대 연결
성분)와 끊어진 섬 클러스터는 고아 검사를 전부 통과하면서 지식 그물 밖에
고립된다. lint가 연결 성분 분석으로 섬을 검출하고, 수리는 원칙 10을 따라
다리 후보(섬↔본토 최고 유사도 페어) 제시까지만 한다 — "의도된 섬"인지의
판단은 사람의 것이다.

- 점검: `canopy lint --json | jq '.counts.island // 0'` == 0 (또는 각 섬이 의도된 것인지 확인됨).

## 비-목표 (하지 않는 것)

- **자동 커밋** — 원칙 5의 이유로. 사용자가 명시적으로 거부한 설계.
- **위키 데이터 형식 발명** — 순수 markdown + frontmatter + wikilink.
  Obsidian이 그대로 열 수 없는 형식은 도입하지 않는다.
- **외부 서비스 의존** — 임베딩·검색·상태 모두 로컬. 네트워크가 필요한 것은
  git push/pull과 모델 최초 다운로드뿐. 웹 UI도 같은 규칙을 따른다:
  하이라이팅은 서버사이드(chroma), mermaid·수식·그래프 라이브러리는 CDN이
  아니라 바이너리에 내장(vendor)되어 오프라인에서 완결된다
  ([web-ui-plan-3.md](web-ui-plan-3.md) D2).
- **TLS 직접 종단** — 전송 암호화는 tailscale·리버스 프록시의 일이다.
  인증(신원)은 canopy가, 암호화(전송)는 네트워크 계층이 맡는다.
